ROUTER
━━━━━━━━━━━━━━━━━━━━━━━━━━

WHAT THIS FOLDER IS:
This folder owns navigation. It decides which page should render for each URL and which routes are public or protected.

AppRouter.jsx defines every URL in the app and which page (component) should show when a user visits or clicks that link.
ProtectedRoute.jsx is a guard that blocks unauthenticated users.

━━━━━━━━━━━━━━━━━━━━━━━━━━
WHY createBrowserRouter INSTEAD OF <BrowserRouter>?
━━━━━━━━━━━━━━━━━━━━━━━━━━

In older React Router (v5) you wrapped everything in <BrowserRouter>:

  <BrowserRouter>
    <Routes>
      <Route path="/" element={<Home />} />
    </Routes>
  </BrowserRouter>

In v6.4+, createBrowserRouter is the modern way:

  const router = createBrowserRouter([...])
  <RouterProvider router={router} />

Why v6.4+ is better:
1. Data loading — routes can have loader functions that fetch data before the component renders (no more loading spinners in components)
2. Error handling — if a page breaks, only that page shows an error, not the whole app
3. Better for big apps — each route clearly defines its URL and what should happen in one place

━━━━━━━━━━━━━━━━━━━━━━━━━━
UNDERSTANDING THE ROUTE CONFIG
━━━━━━━━━━━━━━━━━━━━━━━━━━

const router = createBrowserRouter([
  {
    path: '/',
    element: <AppLayout />,
    children: [
      { index: true, element: <HomePage /> },
      { path: 'shop', element: <ShopPage /> },
      { path: 'cart', element: <CartPage /> },
    ],
  },
]);

── path ──────────────────────────────────────────

path is the URL segment React Router watches for.
Think of it like a street address. When you type
https://tradevo.com/shop into the browser, React Router
reads /shop, finds the matching route, and renders that component.

  { path: 'shop', element: <ShopPage /> }  // visiting /shop → renders <ShopPage />
  { path: 'cart', element: <CartPage /> }  // visiting /cart → renders <CartPage />

Child paths are relative to their parent.
So path: 'shop' inside path: '/' means the full URL is /shop.

── What is '/' ───────────────────────────────────

'/' is the root path — the base URL of your app:

  development  → http://localhost:5173/
  production   → https://yoursite.com/

In the router it's assigned to the layout component (RootLayout),
making it the parent shell of every other route.

Think of it like a folder:

  /              ← the folder (RootLayout)
  ├── index      ← what you see when you open the folder (Home)
  ├── shop       ← subfolder
  ├── cart       ← subfolder
  └── *          ← anything else (NotFound)

── index ─────────────────────────────────────────

index: true marks the default child — the one that renders
when the URL matches the parent exactly, with nothing after it.

  {
    path: '/',
    element: <AppLayout />,
    children: [
      { index: true, element: <HomePage /> },  // renders at exactly /
      { path: 'shop', element: <ShopPage /> }, // renders at /shop
    ],
  }

You can only have one index per parent. It has no path of its own.
Note: index takes a boolean not a string → index: true not index: 'true'

── children and <Outlet /> ───────────────────────

children is how you nest routes inside a parent layout.
React Router renders the parent first, then renders the matching
child at the exact spot where you place <Outlet />.

Think of <Outlet /> like a TV screen in a room.
The room (AppLayout) is always there — navbar, footer, shell.
The screen (<Outlet />) is where the content changes per route.

  function AppLayout() {
    return (
      <>
        <Navbar />
        <main>
          <Outlet />  {/* child page renders here */}
        </main>
        <Footer />
      </>
    );
  }

Without <Outlet /> in the parent, children match but render nothing.

━━━━━━━━━━━━━━━━━━━━━━━━━━
WHEN TO NEST ROUTES
━━━━━━━━━━━━━━━━━━━━━━━━━━

Nest when two or more routes share UI that shouldn't remount.
That shared UI goes in the parent, the changing part goes in <Outlet />.

  shared navbar/footer            → one level of nesting (Tradevo's current setup)
  shared sidebar within a section → two levels of nesting
  totally independent pages       → flat, no nesting

Example — dashboard with its own sidebar:

  createBrowserRouter([
    {
      path: '/',
      element: <RootLayout />,         // navbar + footer
      children: [
        { index: true, element: <Home /> },
        { path: 'shop', element: <Shop /> },
        {
          path: 'dashboard',
          element: <DashboardLayout />, // sidebar
          children: [
            { index: true, element: <Overview /> },
            { path: 'orders', element: <Orders /> },
            { path: 'settings', element: <Settings /> },
          ]
        }
      ]
    }
  ])

  visit /dashboard         → Navbar + Sidebar + Overview + Footer
  visit /dashboard/orders  → Navbar + Sidebar + Orders + Footer
  visit /shop              → Navbar + Shop + Footer (no sidebar)

DashboardLayout also needs its own <Outlet /> for its children.

━━━━━━━━━━━━━━━━━━━━━━━━━━
ROUTE MATCHING IN PRACTICE
━━━━━━━━━━━━━━━━━━━━━━━━━━

Case 1: user visits /
  Router matches → / (parent) → index: true (child)
  Renders: <AppLayout> <HomePage /> </AppLayout>

Case 2: user visits /shop
  Router matches → / (parent) → path: 'shop' (child)
  Renders: <AppLayout> <ShopPage /> </AppLayout>

The parent always renders. Only the child swaps out.

━━━━━━━━━━━━━━━━━━━━━━━━━━
URL PARAMETERS — :slug AND :id
━━━━━━━━━━━━━━━━━━━━━━━━━━

The : prefix tells React Router "this is a variable segment, not a literal word."

  { path: 'product/:slug', element: <ProductDetail /> }

This matches any URL shaped like /product/something:

  /product/nike-air-force-1
  /product/black-leather-jacket
  /product/iphone-15-pro-max

Pull the value out inside the component with useParams:
Recall that you can read values from a navigation using the useParams react hook

  const { slug } = useParams();
  // slug → 'nike-air-force-1'

You can name the param anything — :id, :slug, :productId.
The name is just a label for what you're capturing.

Why slug over id:
A slug is a URL-friendly version of a name — lowercase, hyphens, no special chars.

  "Nike Air Force 1"  →  nike-air-force-1
  "iPhone 15 Pro Max" →  iphone-15-pro-max

  /product/nike-air-force-1   ← readable, SEO friendly
  /product/61f3a2bc9d4e8      ← database ID, means nothing to anyone

Use it to find the matching product from your data:

  const { slug } = useParams();
  const product = products.find(p => p.slug === slug);

PROTECTED ROUTES
━━━━━━━━━━━━━━━━━━━━━━━━━━

ProtectedRoute.jsx checks whether the user is logged in before
rendering a page. If not, it redirects them to /login.

  function ProtectedRoute({ children }) {
    const { user } = useAuth();

    if (!user) {
      return <Navigate to="/login" replace />;
    }

    return children;
  }

Wrap any route that requires authentication:

  {
    path: 'dashboard',
    element: (
      <ProtectedRoute>
        <DashboardPage />
      </ProtectedRoute>
    ),
  }

The replace prop means the redirect replaces the history entry
so the user can't hit the back button to sneak back in.
